Migrate Claude Code chats between Macs

Human-readable companion to the migrate-claude-chats skill. The skill automates this; this page explains what it does and why.

The one trap that breaks everything

Agent view (claude agents) re-hosts a backgrounded chat in the working directory recorded inside the transcript — not where you launched it. A chat copied from another Mac still has that old machine's path baked in (/Users/<olduser>/…), which doesn't exist on the new Mac. So it resumes fine in a terminal (you supply a live folder with cd) but crash-loops in agent view: “Session can't start — source session …” / “session is not responding, restarting it.” The repair rewrites that recorded path. That's the whole game.

How chats are stored

ItemWhere / what
Transcript file~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl
Encoded folder namethe chat's absolute working dir with every non-alphanumeric char → -: re.sub(r'[^A-Za-z0-9]','-', abspath)
Per-line "cwd"the dir agent view respawns the session in — the field the repair rewrites
Send to agent view/bg or the ← left-arrow (identical)

The procedure (in order)

1 · Copy the transcripts to the new Mac

From the old Mac, copy the relevant ~/.claude/projects/<folder>/ directories (or individual .jsonl files) via iCloud Drive, AirDrop, or scp into a staging area. Keep an untouched backup — your safety net.

2 · Re-home each chat to the folder matching where its files now live

Pick the chat's real project directory on the new Mac, compute enc(that-abspath), and put the .jsonl in ~/.claude/projects/<enc>/. To make a chat resume “on top of” a synced project folder, its encoded folder must match that folder's path.

3 · Run the repair (dry-run first)
# PREVIEW — change nothing
python3 scripts/fix_chat_cwd.py \
  --map "/Users/OLDUSER/Claude=>/Users/NEWUSER/Library/Mobile Documents/com~apple~CloudDocs/Sync workspace/Claude" \
  --filter "Sync-workspace-Claude" --dry-run

# APPLY — with a fresh backup
python3 scripts/fix_chat_cwd.py \
  --map "/Users/OLDUSER/Claude=>/Users/NEWUSER/Library/Mobile Documents/com~apple~CloudDocs/Sync workspace/Claude" \
  --filter "Sync-workspace-Claude" \
  --backup-dir ~/chat-migration-backup

It rewrites old→new on every line, drops stale bridge-session markers, remaps dead deeper/worktree cwds up to the nearest existing ancestor, and only writes a file when the result is provably consistent (home cwd exists AND encodes to its own folder). Anything else is left untouched and FLAGGED for you to look at. Pass multiple --map prefixes if chats came from several old locations.

4 · Verify

Confirm the session-ID set still matches your backup (expect zero missing), and that no transcript still holds the old /Users/OLDUSER path or a home cwd that doesn't match its folder. The script's summary plus a quick re-scan covers it.

5 · Resume, then background
cd "/path/to/its/project/folder" && claude --resume <session-uuid>
# wait for load, then at the empty prompt:
/bg          # (or press the ← left-arrow)

Agent view opens with the chat listed. Re-attach later from claude agents with →/Enter. Big transcripts (tens of MB) take 10–20s to load — that's size, not a hang.

Critical ordering: run the repair before you background anything. A live, backgrounded session legitimately contains a bridge-session line — that's running state, and the script strips it. Fix dormant chats only; the copy→fix→resume→bg order avoids the problem naturally.

Gotchas worth remembering

  • /bg vs arrow key is a red herring — both background the session. Agent-view failures are about the recorded cwd.
  • Don't pick a chat by title or size alone — duplicate titles exist; use distinctive last-message content or the session ID.
  • The encoding is lossy — spaces, slashes, and dots all become -. Always derive a folder by encoding a known path, never by decoding the folder name.
  • Subfolder & deleted-worktree cwds are the tricky cases; the script's safety gate catches them and either remaps to a valid ancestor or flags the file rather than guessing.
