# Cowork instructions — the to-do system

What's peculiar about how this system is run. Everything else (script internals, file formats, current task data) lives in the code or in the JSON itself.

---

## What this system is

A personal to-do list spread across multiple Google Calendar calendars (configured in `state_v3.json → calendar_config`). Tasks are consolidated into `tasks_v3.json`, ranked, and surfaced through `check_in.html` (interactive review form) and per-task recommendation pages under `recs/`. The HTML form is where decisions get made; `scripts/pickup_actions.js` applies them back to `tasks_v3.json`.

## Files

| File | Purpose |
|---|---|
| `scripts/tasks_v3.json` | Single source of truth for tasks (lives inside `scripts/` because the v3 generators all read it via `__dirname`). |
| `state_v3.json` | System state — last runs, thresholds, recent notes, landmarks, **calendar configuration**. Top level. |
| `check_in.html` | Interactive review form (`scripts/build_check_in.js`). Primary decision surface. Each task with a solution links to its per-task rec page via "View suggested plan →". |
| `recs/rec_<task_id>.html` | Per-task recommendation pages (one HTML file per active task), generated by `scripts/build_per_task_recs.js`. Rich-HTML rendering (headers/lists/tables/links via `marked`). Linked from `check_in.html`; not meant to be opened directly. |
| `done.md` | Closed-task archive (plain text). |
| `someday_list.md` | Low-urgency parking lot. |
| `inbox.md` | Open questions queue. |
| `gcal_todo_instructions.md` | Calendar query rules — read before touching the calendar. |
| `MAP.md` | Folder map — what each root file is, what's canonical vs. retired. Read first on any session. |
| `scripts/` | Generators, the verb library (`task_verbs.js`), and pickup (`pickup_actions.js`). |
| `archive/` | Frozen historical snapshots. No new content lands here. |

## Calendar gestures (peculiar — read before any closure)

The user signals task state by manipulating calendar events directly. Misreading these gestures is the most likely way to lose work or close the wrong thing.

- **Stacking instead of deleting.** Calendar events are not deleted. Retired/merged events get moved onto a recent whitespace day at a common time slot. The current stacking day lives in `state_v3.json.stacking_day`. "Delete" in conversation means "stack."
- **N+ overlap on a stacking day = "these are done."** When ≥ N non-recurring to-do-shaped events sit at the same time slot on a stacking day, that stack is closure intent. Recurring events do not count toward the overlap. Stacked events do not need to be in `tasks_v3.json` — the gesture is the signal, regardless of tracking. On detection: intersect the stack's titles with `tasks_v3.json` and mark only the intersected entries `done`. Threshold lives in `state_v3.json.thresholds.stacked_closure_threshold` (default 4).
- **Black-colored events = decided not to do.** Treat as inactive/ignored; the event stays in place but the color signals declined.

## Talking in chat — voice rules

These rules govern how to *refer to* tasks in chat. They're surface-behavior rules, not data rules — the JSON / HTML files can use bare `[tNNN]` ids freely (those are stable internal handles); this section is only about what the user reads in a conversation.

- **Always include the task title alongside (or instead of) the `[tNNN]` id.** Bare ids force the user to grep their own files to figure out which task is being discussed. Acceptable: `t059 "Continue drafting follow-up email"` / `"Continue drafting follow-up email" (t059)` / just the title. Not acceptable: bare `t059` or bare `[t059]`. This applies to **confirmation summaries, sweep tables, drain reports, status updates, merge proposals, AskUserQuestion chips, narrative prose** — everywhere a task is mentioned in chat.

- **Always send a confirmation summary after acting** — what was done, which task (title + id), which calendar / file / state changed. The user's receipt and reference.

- **Never alter the user's words** when storing or echoing their original text — verbatim is sacred. Generated *shorthand titles* are okay when the original is too long for a title; in that case the original goes in the description, untouched. Harmless Unicode normalization (smart vs. straight quotes, em-dash vs. hyphen) doesn't count as alteration.

## Wording preciseness + approval workflow

Calendar event titles are precise — they reflect the user's framing, often verbatim from a conversation or message. **Never rephrase a calendar title without explicit approval.** Same rule for descriptions and merges that change content.

Approval flow:
1. Present a table — current title → proposed title (or "no change") + one-line reason.
2. Wait for approval per row.
3. Then execute.

Pure time-slot moves (no content edit) don't need approval.

---

## Merging tasks

When two tasks are essentially the same:
- Keep the one with **more information** as the survivor.
- Update the survivor's description to incorporate the other's info.
- Stack the absorbed task (don't delete).
- Present the merge for approval before editing.

---

## Logical subtasks

If a task naturally contains sub-items (e.g. an umbrella task like "Do all moving-house tasks" → arrange movers, change address, ship boxes), capture sub-items inside the parent's description rather than as separate top-level tasks. Only the umbrella event survives on the calendar; absorbed children get stacked.

`tasks_v3.json` schema supports a `subtasks: [{title, done}]` array. `pickup_actions.js` auto-marks the parent done when all subtasks check.

---

## Ranking philosophy

> urgency (hard deadlines, blocking others) > time-decay (professional emails go stale) > importance for active projects > logistics with deadlines > health > professional comms > personal/admin > leisure

## Someday vs. main list

Low-urgency / "interested but not now" items go in `someday_list.md`, not the main task list. Think reading-list, not to-do.

---

## Decisions go through the HTML form, not docx comments

An older version of this system used inline Word comments on `to do.docx` as the channel for "Claude, do X to this task." That mechanism is deprecated. The current answer surface is the form in `check_in.html`: radios + textarea per task, with `pickup_actions.js` applying decisions back to `tasks_v3.json`. If you encounter old docx comments in `archive/`, treat them as historical record, not instructions.

## File format conventions

- **HTML** for interactive review surfaces (`check_in.html`) and for per-task recommendation pages (`recs/rec_<id>.html`).
- **Markdown** for plain-text reads (`inbox.md`, `someday_list.md`, `done.md`, `MAP.md`, `cowork_instructions.md`, rolling `check_in_YYYY_MM_DD.md`).
- **JSON** for structured state read by scripts (`state_v3.json`, `scripts/tasks_v3.json`).
- **No new `.docx` or `.pdf`.** Existing docx files in `archive/` stay as frozen historical record; nothing new gets generated in those formats.

## Suggested-plan quality (load-bearing acceptance criteria)

The "Suggested plan" attached to each task (`solution` field, rendered into `recs/rec_<task_id>.html` by `scripts/build_per_task_recs.js`) is the make-or-break feature of this system. Common failure modes: plans that feel overwhelming, plans that are made up rather than researched, plans rendered as wall-of-text prose.

Treat these as acceptance criteria. A plan that doesn't meet them is "needs review," not "done":

- **Researched, not made up.** Pull from real sources — not Claude's training memory alone. Reach for installed research-grade skills/tools when relevant: `research-27` (citation-rich brief, APA refs), `citation-deepening` (verbatim source quotes), `source-quality-check` (rate venue tier / peer review), `WebSearch` + `WebFetch` for live URL / price / form-version checks. If a recommendation includes a date-sensitive fact (deadline, price, contact, policy), verify it against a live source in the same pass — do not rely on memory.
- **Search the user's own data when relevant.** Free to query Gmail, Drive, Calendar via MCP when the task plausibly has prior context. Example: emailing a professor → search inbox for past correspondence even if the user doesn't expect any. The freedom to look is the point.
- **Length matches usefulness; clean over comprehensive.** If opening the plan feels like work, it failed. Default to the smallest piece that unblocks the next action. Cut hedging caveats, multi-option comparison tables that aren't decision-relevant, and "background" sections that don't change the action. A rec with 3 sharp steps beats a rec with 8 well-researched ones.
- **Write the `solution` field in RICH markdown — not dry prose.** `build_per_task_recs.js` runs `marked` over it, so the rendered page gets proper headings (`##`, `###`), lists (`- `, `1. `), tables (`| col | col |`), links (`[label](url)`), bold/italic, and code blocks. **Plain prose with no structure renders as a wall of `<p>` tags.** Aim for the rec page to feel like a polished short brief, not a paragraph dump. Concretely: open with a one-line status / verdict, use `##` for each step or section, bullets for sub-steps, tables for comparisons, links for every claim that has a source.
- **Cite what you used.** Either as inline `[label](url)` markdown links (preferred — they render in the body) and/or as a short "## Sources" section at the end. The `solution_sources` array on the task is also rendered as a footer list if populated.
- **Open-question handling.** Use `solution_questions` for blocking questions; they render in a callout box on the rec page. Don't bury questions in prose where the user will miss them.

When a rec page renders empty or as prose-only on opening, that's a generator regression — the fix is to rewrite the `solution` field in structured markdown, not to add styling.
